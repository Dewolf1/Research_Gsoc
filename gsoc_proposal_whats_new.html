The biggest upgrade. The old proposal invented pain points from documentation. The new one quotes the people who actually said them in the thread. Here's what was added:

Search backends silently overwrite each other
WAGTAILSEARCH_BACKENDS configs overwrite each other for default search instead of matching the active locale. It's impossible to have locale-specific NLP in the search.
— enzedonline, wagtail/wagtail #13693
Confirmed by issue #1177. Now cited directly rather than described in the abstract.
localized silently returns the wrong locale
localized and localized_draft methods return the source object if no translation is found in the active locale. You often want this to return None or throw an error.
— enzedonline, wagtail/wagtail #13693
The same contributor published working localized_or_none code. The proposal now acknowledges this and says the API belongs in core — not reinvented from scratch.
RFC 54 design intent (the Belgian French case)
Some sites create new translations from other translations. For example, you might translate English to French, then localise for Belgium rather than translate from English again. This would mean the French version is both a translation and a source.
— kaedroho (wagtail-localize maintainer), wagtail/wagtail #13693
The old proposal hand-waved past this. The new one quotes it directly and explains why Part 4 uses a heuristic approach documented with explicit limitations.
Draft sync doesn't work — out of scope, explained why
The top customer complaint is that the sync does not work. This usually happens when the customer wants to prepare a change in draft in both the source and the translations.
— th3hamm0r, wagtail/wagtail #13693
This problem is real and serious — but it belongs in wagtail-localize, not core. The proposal now explicitly names the issue and explains the scope decision, rather than silently ignoring it.

The old proposal described what to build. The new one sketches how. Here are the key technical upgrades:

Part 1 — Sitemaps: x-default is now included
The old version only added hreflang alternates. The new code sketch also appends an x-default entry pointing to the default-locale variant — which Google explicitly requires.
alternates.append({ 'lang_code': 'x-default', 'location': (default_url or self).full_url, })
Part 2 — localized_or_none: DraftStateMixin check added
The old version had a vague sketch. The new one handles the pages-vs-snippets distinction: for page models that use DraftStateMixin, the method checks .live before returning the result — matching how enzedonline's production code works.
if result and isinstance(self, DraftStateMixin) and not result.live: return None
Part 4 — Source identity: honest about the limitation
The old version claimed these would be "read-only properties derived by walking the translation tree." The new one is explicit that is_source_translation is a heuristic based on the default locale — and documents exactly when it fails.
""" Heuristic: True if this object's locale matches the site default. This is NOT a guarantee. RFC 54 intentionally does not track source/translation relationships in core. For accurate tracking, use wagtail-localize's TranslationSource model. """
Part 5 — Search: concrete config shape defined
The old version said "implement a scoped configuration matrix." The new one shows exactly what the config would look like, matching enzedonline's own suggestion in the thread.
WAGTAILSEARCH_BACKENDS = { 'default': { 'BACKEND': '...', 'SEARCH_CONFIG': 'english', # fallback 'LOCALES': { 'fr': {'SEARCH_CONFIG': 'french'}, 'de': {'SEARCH_CONFIG': 'german'}, } } }
