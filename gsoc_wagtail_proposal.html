Multilingual Support Improvements for Wagtail
Mohd Adeeb New Delhi, India · UTC+5:30 350 hours · Large project Mentors: Thibaud Colas, Coen van der Kamp
In plain words

Wagtail is a popular Django CMS used by many teams to run websites in multiple languages. The core framework is well-designed — but over years of real use, developers have repeatedly hit the same missing pieces. This proposal fills five specific gaps that cause real pain in production today: broken SEO, missing APIs, and configuration headaches. Every change is additive — nothing existing breaks.

The five problems being solved
01
Sitemaps don't list translated pages
Google can't find your French or Arabic versions — and penalises you for duplicate content.
02
Silent wrong-language fallback
.localized returns the source object when no translation exists — with zero warning.
03
Settings can't be localised
You can't store different contact emails or footer text per language without hacky workarounds.
04
No way to tell source from translation
has_translation returns True even on translated items — the name is misleading.
05
Search ignores active language
French search can't use French NLP rules — all locales share the same search config.
Technical implementation
1
Multilingual sitemaps
hreflang alternates + x-default support
High confidence
▼
What & why in plain English
A sitemap is a file that tells Google which pages your site has. Right now, Wagtail's sitemap only lists the default-language pages. If you have a French translation of a page, Google never learns it exists — so it treats the French URL as duplicate content and penalises your rankings.

The fix: when i18n is on, attach hreflang links to every page — these are small XML tags that say "the French version of this page is at /fr/…". Also add an x-default entry that tells search engines which version to show by default.
"The sitemap only produces a list for pages in the default locale. It lacks hreflang entries and the x-default value."
— enzedonline, Wagtail #13693
# wagtail/models/pages.py def get_sitemap_urls(self, request=None): urls = super().get_sitemap_urls(request) if getattr(settings, 'WAGTAIL_I18N_ENABLED', False): alternates = [] # Fetch all live translations of this page qs = (self.get_translations(inclusive=True) .live() .defer_streamfields() # skip heavy stream data .select_related('locale')) for page in qs: alternates.append({ 'lang_code': page.locale.language_code, 'location': page.full_url, }) # x-default = point search engines to the default-locale version default = self.get_translation_or_none(Locale.get_default()) alternates.append({'lang_code': 'x-default', 'location': (default or self).full_url}) urls[0]['alternates'] = alternates return urls
All translated pages indexed by Google No more duplicate-content penalties x-default for default locale
2
localized_or_none
Explicit API — no silent wrong-language fallback
High confidence
▼
What & why in plain English
When you call .localized on a Wagtail object and no translation exists for the active language, it silently gives you the source object back — in the wrong language — with no warning at all. Your French site renders English content invisibly.

The fix is a new property: .localized_or_none. It does the same lookup but returns None when nothing is found, so your code can handle the missing translation explicitly — show a fallback UI, log a warning, skip the block. The old .localized is untouched.
# wagtail/models/i18n.py — two new properties on TranslatableMixin @property def localized_or_none(self): """Like .localized but returns None when no translation exists. Never silently shows the wrong language.""" if not getattr(settings, 'WAGTAIL_I18N_ENABLED', False): return self locale = Locale.get_active() if locale.id == self.locale_id: return self result = self.get_translation_or_none(locale) # For pages: also check it's published, not just a draft if result and isinstance(self, DraftStateMixin) and not result.live: return None return result @property def localized_draft_or_none(self): """Same but includes unpublished drafts — useful in the admin.""" ...
Explicit control over missing translations No migration needed Old .localized unchanged
3
Translatable settings
Per-locale configuration for BaseSiteSetting & BaseGenericSetting
Medium confidence
▼
What & why in plain English
Wagtail's Settings models let site editors store global config — things like a contact email, footer text, social media links, or a phone number. But right now these settings have no concept of language. Every locale sees the same values.

The fix: apply BootstrapTranslatableMixin to the settings models (only when i18n is enabled), add locale-aware lookup with a fallback to the default locale, and add a locale switcher in the settings UI — exactly like the one already on the page editor. A monolingual install sees zero schema changes.
Different contact info per language Per-locale footer text, links Locale switcher in admin UI Fallback to default locale Zero impact on monolingual installs
4
Source identity heuristics
Tell apart source objects from their translations
Medium confidence
▼
What & why in plain English
Wagtail currently has a method called has_translation, but it returns True even on translated items — because it checks if any translation exists, not whether this item is the original. Confusing and misleading.

The fix adds two helpers. is_source_translation is a heuristic (not a guarantee): it returns True if this object's locale is the site default. get_translation_siblings() returns all objects sharing the same translation key — useful for writing safe pre_delete signal handlers that clean up related translations. Both include clear docstrings warning about the Belgian-French edge case.
@property def is_source_translation(self): """ Heuristic only — True if this object's locale is the site default. Edge case: French → Belgian French is valid (RFC 54 does not track source/translation in core). For accurate tracking use wagtail-localize. """ return self.locale_id == Locale.get_default().pk def get_translation_siblings(self): """All objects sharing this translation_key across locales. Use in pre_delete signal handlers to decide what to do with orphaned translations when a source is deleted.""" return self.__class__.objects.filter( translation_key=self.translation_key ).select_related('locale')
Safe delete-signal handlers Clear docstrings for edge cases No RFC 54 design changes
5
Locale-aware search config
Per-language NLP dictionaries for Wagtail search
Deferral candidate
▼
What & why in plain English
Wagtail uses PostgreSQL's full-text search, which supports language-specific stop-words and stemming — e.g., French "rechercher" and "recherche" match the same root. But right now there's no way to say "use French config when the active locale is French", because all locales share one SEARCH_CONFIG.

The fix introduces an optional LOCALES sub-key in the search backend config. Existing configs without LOCALES are completely unchanged. A new helper get_search_backend_for_locale() merges overrides onto the base config at query time. If this part overruns, it's safely deferred — Parts 1–4 still form a complete, independently mergeable contribution.
# settings.py — fully backward compatible, LOCALES key is optional WAGTAILSEARCH_BACKENDS = { 'default': { 'BACKEND': 'wagtail.search.backends.database', 'SEARCH_CONFIG': 'english', # fallback for unmatched locales 'LOCALES': { 'fr': {'SEARCH_CONFIG': 'french'}, 'de': {'SEARCH_CONFIG': 'german'}, } } } # New helper — merges locale overrides onto the base config backend = get_search_backend_for_locale('fr') # gets french config backend = get_search_backend_for_locale('ja') # falls back to english
Locale-specific NLP dictionaries Old get_search_backend() unchanged Named deferral candidate
